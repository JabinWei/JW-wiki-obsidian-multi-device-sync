> 本文档定义本 Obsidian 仓库的**笔记编写、组织与维护的默认规则**。所有新建笔记均遵循此规范，确保知识库长期可维护。
>
> 此文档为**活文档**——规则随实践持续迭代。

## 1. 顶层目录分层（IAR 框架）

```
obsidian-notes/
├── 00-Core/         核心层 — 长期沉淀的知识（知识库的主体）
├── 10-Action/       行动层 — 待办任务与行动项
├── 20-Inspiration/  灵感层 — 收集的外部素材与启发
├── 30-Time/         时间层 — 计划、回顾与日志
└── 90-Storage/      存储层 — 模板、资源、归档
```

| 层级 | 职责 | 存放内容 |
|------|------|----------|
| **00-Core** | 知识主体 | 系统学习笔记、原子卡片、MOC 索引、项目文档 |
| **10-Action** | 行动管理 | 待办事项、任务清单 |
| **20-Inspiration** | 灵感收集 | 外部素材、摘录、启发性的非原创内容 |
| **30-Time** | 时间管理 | 愿景规划、周计划、周期回顾 |
| **90-Storage** | 基础设施 | 模板、图片、附件、归档笔记 |

> **核心原则**：知识沉淀进 00-Core，行动管理进 10-Action，临时灵感进 20-Inspiration。

## 2. 00-Core 子目录结构

```
00-Core/
├── 01-CoreKnowledge-核心知识/    系统化学习笔记（长文）
│   └── {C分类}/ {CB分支}/ {PL领域}{序号}-{标题}.md
├── 02-CardKnowledge-卡片知识/    原子化知识卡片（短文）
│   └── {领域前缀}-{序号}-{标题}.md
├── 03-KnowledgeMap-知识地图/     MOC 索引（链接中枢）
└── 04-Projects-项目产出/        项目文档与产出
    └── {PJ项目}/ {OB文档}{序号}-{标题}.md
```

| 子目录 | 笔记类型 | 粒度 | 模板 |
|--------|----------|------|------|
| 01-CoreKnowledge | 系统学习笔记 | 大（完整主题） | TP011 |
| 02-CardKnowledge | 原子卡片 | 小（单一概念） | TP002 |
| 03-KnowledgeMap | MOC 索引 | 链接地图 | TP003 |
| 04-Projects | 项目文档 | 项目级 | TP004 |

> **区分原则**：
> - **核心知识**：系统学习一个主题，有前置知识、学习目标、练习验证 → `type: learning`
> - **卡片知识**：深入一个独立概念，足够原子化可被多处引用 → `type: card`
> - **知识地图**：不做知识承载，只做索引链接 → `type: moc`

## 3. 编号前缀体系

### 3.1 分类代码

| 前缀 | 全称 | 含义 |
|------|------|------|
| **C** | Category | 知识大类 |
| **CB** | Category Branch | 知识分支 |
| **PL** | Programming Language | 编程语言 |
| **MA** | Math | 数学 |
| **EN** | English | 英语 |
| **ES** | English Speaking | 英语口语 |
| **PY** | Python | Python 语言 |
| **TP** | Template | 模板 |
| **PJ** | Project | 项目 |
| **OB** | Obsidian | Obsidian 相关 |

### 3.2 文件命名规则

**核心知识**：
```
格式：{领域前缀}{序号}-{中文标题}.md
示例：PY005-Python中的变量.md
      CL001-函数.md
```

**卡片知识**：
```
格式：{领域前缀}-{序号}-{中文标题}.md
示例：PL-001-编程语言的变量与类型系统.md       ← 通用编程概念
      PL-PY-001-is与==的区别.md               ← Python 专属概念
      PL-003-哈希值底层原理.md                 ← 通用编程概念
```

> **卡片编号规则**：
> - `{领域}-{序号}`：跨语言的通用概念（如 PL-001, PL-002, PL-003...）
> - `{领域}-{语言}-{序号}`：特定语言的概念（如 PL-PY-001, PL-PY-002...）

**项目产出**：
```
单文件：{项目前缀}{序号}-{中文标题}.md
示例：OB001-UPDATE LOG.md
      OB002-Obsidian仓库配置文档.md
      OB003-仓库笔记管理默认规则.md
      PJ002-知识卡片提取计划.md

文件夹（定期报告/多文件项目）：
  {项目前缀}{序号}-{主题}/
  └── YYYY-MM-DD-{报告标题}.md
示例：PJ003-仓库健康报告/
      └── 2026-07-20-仓库健康报告.md
```

**模板**：
```
格式：TP{序号}-{中文描述}.md
示例：TP001-通用笔记模版.md
      TP002-卡片笔记模版.md
```

## 4. Frontmatter 规范

### 4.1 通用字段

| 字段 | 必需 | 说明 | 示例 |
|------|------|------|------|
| `type` | ✅ | 笔记类型 | `card`, `learning`, `note`, `moc`, `project` |
| `status` | ✅ | 完成状态 | 见 §4.2 |
| `tags` | ✅ | 标签列表 | `[编程语言]`, `[Python, 编程语言]` |
| `created` | ✅ | 创建日期 | `2026-06-23` |
| `updated` | ✅ | 更新日期 | `2026-06-23` |

### 4.2 状态（status）取值

| 状态值 | 适用类型 | 含义 |
|--------|----------|------|
| `已完成` | card, learning | 内容完整，无需补充 |
| `待办` | card, learning | **主体已完成**，但标注了知识缺口，需后续迭代补充 |
| `学习中` | learning | 正在系统学习，内容持续更新 |
| `草稿` | note | 临时草稿，尚未结构化 |
| `planning` | project | 项目规划阶段 |

**状态流转**：

```
草稿 ──→ 待办 ──→ 已完成
  │                 ↑
  └──→ 学习中 ──────┘
```

> **`待办` 的精确含义**：笔记的**核心内容已写入**，结构完整、可独立阅读，但笔记底部 `📝 待补充内容` 清单中标注了尚未覆盖的知识点。这是「已完成」和「未开始」之间的中间态——**可用但未完成**。

### 4.3 类型专属字段

**卡片笔记（card）**：

| 字段 | 必需 | 说明 |
|------|------|------|
| `aliases` | 推荐 | 别名列表，用于 Obsidian 搜索和双链引用 |

```yaml
---
type: card
status: 待办
aliases: [哈希值, Hash, hashable, 可哈希, 散列]
tags: [编程语言]
---
```

**学习笔记（learning）**：

| 字段 | 必需 | 说明 |
|------|------|------|
| `domain` | 推荐 | 所属领域 |

```yaml
---
type: learning
status: 学习中
domain: Python
tags: [Python, 编程语言]
---
```

**项目笔记（project）**：

| 字段 | 必需 | 说明 |
|------|------|------|
| `priority` | 可选 | `high`, `medium`, `low` |
| `start_date` | 可选 | 项目开始日期 |
| `due_date` | 可选 | 截止日期 |

### 4.4 标签（tags）规范

- 使用**中文标签**作为主分类：`编程语言`、`数学`、`英语`、`obsidian`
- 语言专属内容附加语言标签：`[编程语言, Python]`
- 项目文档附加项目标签：`[obsidian, 配置文档, 项目产出]`
- 标签用于 Dataview 查询和 Obsidian 标签面板聚合

## 5. 各类型笔记结构规范

### 5.1 卡片笔记（type: card）→ TP002

```markdown
## 💡 核心概念       ← 一句话定义，加粗关键术语
## 🔍 详细说明       ← 分节展开（### 一、### 二、...）
## 📎 例子           ← 至少一个可运行代码示例
## ⚡ 反例 / 边界     ← 常见误区、局限性、不适用场景
## 🔗 关联卡片       ← [[双向链接]] 到相关卡片
## 📥 被哪些笔记引用  ← Dataview 查询块
```

> **要求**：每张卡片一个概念，足够原子化。例子优先用 Python，兼顾其他语言。

### 5.2 系统学习笔记（type: learning）→ TP011

```markdown
## 🎯 学习目标       ← 学完后应能回答的问题（问题列表）
## 📖 前置知识       ← [[前置笔记链接]]
## 📚 核心内容       ← 笔记主体（### 概念1, ### 概念2...）
## 🧪 练习 / 验证    ← 具体练习题 + 答案
## 🤔 常见误区       ← 典型错误认知 + 反例代码
## 🔗 相关资源       ← 上一节/下一节 + 外部链接
```

> **要求**：有明确的学习目标和练习，上下节用 `[[链接]]` 串联形成学习路径。

### 5.3 MOC 索引（type: moc）→ TP003

```markdown
## 主题概述         ← 一段话概括领域范围
## 核心概念         ← 关键概念卡片列表
## 子主题           ← 分层展开 + 笔记链接
## 学习路径         ← 建议的阅读顺序
## 关联领域         ← 交叉引用其他 MOC
```

> **要求**：不做知识承载，只做导航链接。保持轻盈，避免内容膨胀。

### 5.4 项目笔记（type: project）→ TP004

```markdown
## 🎯 项目目标       ← 一句话说清要达成什么
## 🔧 当前状态       ← 进展到哪一步，卡在哪里
## 📋 任务清单       ← 待开始 / 进行中 / 已完成
## 🚧 卡点 / 风险    ← 当前阻碍
## 📝 过程记录       ← 按时间线的关键决策日志
## 📎 相关资料       ← 关联笔记 + 外部链接
## 🔄 复盘           ← 项目结束后填写
```

### 5.5 通用笔记（type: note）→ TP001

用于尚未归类的临时笔记，结构自由。

## 6. 内容编写规则

### 6.1 链接规则

- 统一使用 Obsidian **wikilink** 格式：`[[笔记名称]]`
- 链接到具体章节：`[[笔记名称#章节标题]]`
- 链接别名：`[[笔记名称|显示文字]]`
- **内联优先**：关联链接应嵌入笔记正文中概念首次出现或展开讨论的位置，而非仅堆积在文末 `🔗 关联卡片` 节。文中链接提供阅读时的即时跳转入口，文末章节负责汇总追溯。
- 关联卡片部分只放**有直接逻辑关系**的链接，不做大而全的链接堆砌

### 6.2 引用块

- 关键结论使用 `> 🔑` 引用块突出
- 注意事项使用 `> ⚠️` 引用块
- 补充说明使用 `> 💡` 引用块

### 6.3 代码块

- 必须标注语言：` ```python `, ` ```java `, ` ```javascript `
- 反例代码优先在注释中标注 `❌`，正例标注 `✅`

### 6.4 表格

- 对齐使用 Obsidian 的管道符表格
- 对比类信息优先用表格呈现

### 6.5 Dataview 查询

- 每张卡片底部固定使用以下查询块追溯引用：

  ````markdown
  ## 📥 被哪些笔记引用

  ```dataview
  LIST
  FROM [[]]
  SORT file.name ASC
  ```
  ````

## 7. 待办标记约定（新增）

### 7.1 `status: 待办` 的使用场景

笔记的**主体内容已写入**，但存在以下任一情况时标记为 `待办`：

1. 某些子主题尚未覆盖（有明确的知识缺口）
2. 部分示例/练习尚未编写
3. 需要后续补充外部参考资料
4. 相关知识的深度扩展尚未完成

### 7.2 待补充内容清单

标记为 `待办` 的笔记，必须在文末添加 `📝 待补充内容` 节，使用 Markdown 任务列表：

```markdown
## 📝 待补充内容

- [ ] 主题 A — 简要说明尚缺什么
- [ ] 主题 B — 简要说明尚缺什么
```

### 7.3 生命周期

```
笔记创建 → status: 待办 + 📝 待补充清单
    ↓
补充完所有缺口 → status: 已完成，移除 📝 节
    ↓
发现新缺口 → status 重新设为 待办，追加 📝 条目
```

> **区别于 `学习中`**：`学习中` 表示**整个主题**正在学习，内容仍在快速迭代；`待办` 表示笔记结构已经稳定，但存在**可枚举的知识盲区**，适合后续定向填补。

## 9. 文件操作规则

### 9.1 新建笔记

1. 确定笔记类型（card / learning / moc / project / note）
2. 使用对应模板
3. 按编号规则分配文件名
4. 补全 frontmatter（type, status, tags, aliases, created）

### 9.2 修改已有笔记

1. 更新 frontmatter 中的 `updated` 日期
2. 若新增知识缺口，将 `status` 改回 `待办` 并追加 `📝 待补充内容`
3. **变更日志记录**（仅限 `status: 已完成` 的笔记）：
   - **记录条件**：笔记内容发生实质性变更（新增章节、补充知识点、修改示例代码、更正错误等）
   - **不记录**：纯元数据变更（编号调整、链接更新、文件重命名等）
   - **记录方式**：
     1. 在 `90-Storage/97-Changelog-更新日志/` 目录下创建/编辑 `CHGLOG-[笔记文件名].md`
     2. CHGLOG 文件顶部写入 `> 关联笔记：[[原笔记名]]` 建立反向链接
     3. 在原笔记 `## 🔗 相关资源` 节中追加 `- 📝 变更日志：[[CHGLOG-xxx]]` 建立正向链接
     4. 表格格式：`| 日期 | 变更内容 |`
   - **首次记录**：若 `CHGLOG-[笔记文件名].md` 尚不存在，则新建文件

### 9.3 归档

- 不再活跃但仍有参考价值的笔记 → `90-Storage/91-Archive-归档/`

### 9.4 Git 同步

**每次修改后必须同步远程仓库**，确保多设备可用且历史可追溯：

```bash
git add -A
git commit -m "<type>: <简短描述>"
git push origin master
```

> **提交信息规范**：使用约定式提交前缀 — `feat`（新卡片/新功能）、`fix`（修复）、`docs`（文档更新）、`refactor`（结构调整）、`chore`（杂项）。

## 10. AI 辅助书写约定

当 AI（Claude Code）协助创建或编辑笔记时，遵循以下约定：

1. **遵循本规则文档**的所有命名、结构和 frontmatter 规范
2. **默认状态**：新卡片笔记默认 `status: 待办`，底部附 `📝 待补充内容` 清单，标注 AI 未覆盖的知识点
3. **标签**：根据内容自动分配中文标签（编程语言 / 数学 / 英语 / obsidian 等）
4. **关联卡片**：自动搜索相关笔记并建立双向链接
5. **示例语言**：代码示例优先使用 Python，概念性内容可多语言对比
6. **卡片知识提取流程**：
   - **判断依据**：源笔记已讲透的跳过，有增量价值（跨笔记整合、深度补充、高频陷阱）的才提取
   - **核心笔记先行**：先检查源笔记是否需要瘦身（去除卡片提取后可精简的重叠内容）
   - **生成预览**：展示给用户确认，标注与源笔记的对比（哪些是新增、哪些是引用）
   - **确认后写入**：用户确认无误后创建卡片文件
   - **补双向链接**：在源笔记正文对应知识点位置加 `[[卡片链接]]`，底部「相关资源」追加一行
   - **更新项目文档**：在 PJ002 标记对应条目为已完成
7. **核心笔记 vs 卡片知识分工**：
   - **核心笔记** = 完整教程，保留基础用法、完整示例、练习
   - **卡片知识** = 深度辨析、横向对比、常见陷阱、跨笔记整合
8. **每次修改后提交远程**：`git add -A && git commit && git push`
