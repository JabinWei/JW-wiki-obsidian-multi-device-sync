# PL-006-浮点数精度误差的二进制根源

## 💡 核心概念

在任何主流编程语言中运行 `0.1 + 0.2`，得到的结果都不是精确的 `0.3`。这不是 Python 的 bug，而是所有使用 IEEE 754 浮点标准的语言的共同行为。

---

## 🔍 详细说明

### 十进制 vs 二进制：分数的表示能力不同

十进制中 1/3 = 0.3333... 是无限循环小数。同理，十进制中很"干净"的 0.1，在二进制中却是无限循环小数：

0.1(十进制) = 0.000110011001100...(二进制)

| 步骤 | 操作 | 结果 |
|------|------|------|
| 1 | 0.1 × 2 = 0.2 | 整数部分 0，小数 0.2 |
| 2 | 0.2 × 2 = 0.4 | 整数部分 0，小数 0.4 |
| 3 | 0.4 × 2 = 0.8 | 整数部分 0，小数 0.8 |
| 4 | 0.8 × 2 = 1.6 | 整数部分 1，小数 0.6 |
| 5 | 0.6 × 2 = 1.2 | 整数部分 1，小数 0.2 |
| 6 | 0.2 × 2 = 0.4 | 回到步骤 2，**开始循环** |

### IEEE 754 如何存储

IEEE 754 双精度（Python 的 `float` 类型）用 64 位存储：

| 部分 | 位数 | 含义 |
|------|------|------|
| 符号位 | 1 bit | 正负号 |
| 指数 | 11 bits | 2 的指数（偏移 1023） |
| 尾数 | 52 bits | 有效数字（隐含前导 1） |

53 个有效位（52 显式 + 1 隐含）之后的部分只能**截断**，这就是精度误差的来源。

---

## 📎 两种解决方案

### Decimal（精确十进制运算）

适用于金融计算等不容许任何误差的场景：

```python
from decimal import Decimal

print(Decimal('0.1') + Decimal('0.2') == Decimal('0.3'))  # True
```

> ⚠️ 必须用字符串 `'0.1'` 传入 Decimal，否则误差已经产生。

### 容差比较（Epsilon Comparison）

适用科学计算等可容忍微小误差的场景：

```python
a, b = 0.1 + 0.2, 0.3
epsilon = 1e-9
print(abs(a - b) < epsilon)  # True
```

---

## ⚡ 反例 / 边界

1. **能用二进制精确表示的分数**（分母是 2^n）没有误差，如 0.5、0.25、0.125
2. **Decimal 比 float 慢数十倍**，需根据场景选择
3. **所有语言共享 IEEE 754**：Python、JavaScript、Java、C/C++、Go 的 `float`/`double` 行为一致
