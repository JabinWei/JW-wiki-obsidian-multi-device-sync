## 💡 核心概念

Python 的 `dict` 和 `set` 底层都是**哈希表（Hash Table）**。哈希表将「查找」变成「计算」——通过 `hash(key) % table_size` 算出存储位置，实现 O(1) 查找。

但选择哈希表作为数据结构**衍生出一个硬性约束**：元素必须是**可哈希的（hashable）**——即哈希值在生命周期内不变，且支持相等比较（`__hash__` + `__eq__`）。

```
哈希表机制                     可哈希约束
──────────                    ──────────
槽位 = hash(key) % table_size → 要求 hash(key) 稳定不变
冲突时用 __eq__ 做最终判定    → 要求 hash(a)==hash(b) 当 a==b
空槽是探测的终止符            → 表永远留 ≥1/3 空槽
```

> 🔑 **两个核心命题**：① 哈希表用空间换时间，O(1) 靠「预留空槽 + 控制负载因子」保证。② 可哈希不是可选项——它是选择哈希表这个数据结构时，物理上必须满足的前提。

## 🔍 详细说明

### 一、集合为什么查找快——「计算」而非「遍历」

**类比：图书馆找书**

- **列表查找（O(n)）**：书按购买顺序放在书架上，从第一本开始一本本翻。
- **集合查找（O(1)）**：书按分类号放在固定位置，直接算出「第几排第几本」，走过去就拿。

**查找动作拆解**——检查元素 42 是否在集合中：

```python
# 1. 计算哈希值
# 42 → hash(42) → 1357924680

# 2. 映射到数组索引
# 1357924680 % table_size → 索引 3

# 3. 直接去索引 3 检查
# - 空 → 不在（O(1)）
# - 存的就是 42 → 在（O(1)）
```

三步完成，与集合大小无关。而列表必须从头逐个比对——这就是 O(1) vs O(n) 的来源。

```
集合 (10 个)   █              ← 1 步
集合 (100 万个) █              ← 还是 1 步
列表 (10 个)   ██████         ← 最多 10 步
列表 (100 万个) ██████████████ ← 最多 100 万步
```

### 二、列表为什么不用哈希表——「能不能」vs「要不要」

列表用 O(n) 线性查找不是因为「元素不可哈希」，而是**设计目标不同**的主动取舍：

| | 列表（动态数组） | 集合（哈希表） |
|------|------|------|
| **核心目标** | 维护顺序、支持索引 | 去重、快速成员检查 |
| **索引访问** | `list[3]` O(1) | ❌ 不支持 |
| **成员检查** | `x in list` O(n) | `x in set` O(1) |
| **内存布局** | 紧凑连续 | 稀疏分布（保留空槽） |

**因果方向**：设计目标 → 数据结构选择 → 约束条件（不是反过来）。

```
集合目标：按值快速查找 → 选哈希表 → 衍生约束：元素必须可哈希
列表目标：有序、按下标访问 → 选数组 → 不要求可哈希 → 按值查找 O(n)
```

### 三、空间换时间——哈希表的本质

预留的空槽就是花空间买的走廊：

```
数组（利用率 100%）：[A, B, C, D, E, F, G, H]
→ 找 G，从头一个一个比，O(n)

哈希表（利用率 ~67%）：[ , B,  , A,  ,  , C,  ]
→ hash("C") → 6 → buckets[6] → C，一步到位，O(1)
```

**越空，碰撞越少，查找越快。** 留空槽 = 多买内存不放东西 = 换取查找时少走路。

> 存满的仓库 vs 预留走廊的仓库：货物塞满整个仓库（100%），找一件得翻遍所有货架。留 30% 空间当走廊，走到坐标直接拿货。

**冲突解决策略**：

| 策略 | 原理 | 代表 |
|------|------|------|
| **拉链法** | 每个槽位维护链表/红黑树（链长 ≥ 8 转红黑树） | Java HashMap |
| **开放寻址法** | 冲突时按探测序列找下一个空位 | Python dict/set |

### 四、为什么有哈希碰撞还是 O(1)

O(1) 指**期望/平均**，不是「恰好 1 步」，而是**步数上限不随 n 增长**。

当负载因子被控制住时，每个桶的平均链长是常数——n=100 平均走 1.5 步，n=1 亿也平均走 1.5 步。步数不变 = O(1)。

| 场景 | 复杂度 | 说明 |
|------|--------|------|
| 无碰撞 | O(1) | ~99.9% |
| 一次碰撞 | O(2) ≈ O(1) | ~0.1% |
| 全部堆一个槽 | O(n) | 极低（除恶意构造） |
| **平均/期望** | **O(1)** | ✓ |

**碰撞时的匹配机制**：槽位存了完整哈希值。先比哈希值（整数比较，极快），不匹配直接跳过；匹配了才比 key 本身：

```
查找 "cat"（hash = 0x7a3f）：
槽位 3：[hash=0x9b2c, key="dog"] → 哈希不匹配 ✗ → 跳过
槽位 4：[hash=0x7a3f, key="cat"] → 哈希匹配 ✓ → key 匹配 ✓ → 命中
```

### 五、负载因子——O(1) 的量化保证

```
负载因子 (α) = 已占用槽位数 / 表的总槽位数
```

α 越小 → 空槽多 → 碰撞少 → 快但费内存。α 越大 → 省内存但碰撞激增。

| 策略 | 能否 α=1.0 | 后果 |
|------|-----------|------|
| **开放寻址法**（Python） | 绝不允许 | 插入失败，查找退化为 O(n) |
| **链表法**（Java） | 可 >1.0 | 链表过长，性能线性下降 |

```
成功查找 ≈ (1/2) × (1 + 1/(1-α))
失败查找 ≈ (1/2) × (1 + 1/(1-α)²)

α=0.5 → 成功 ~1.5 次，失败 ~2.5 次
α=0.9 → 成功 ~5.5 次，失败 ~50 次  ← 越空越快
α=1.0 → 失败 ∞ 次                   ← 表满，没有空槽，探测无刹车
```

Python 在 α≈2/3 时触发扩容——表翻倍，空槽重现，每个桶的平均元素数重新变成与 n 无关的常数。扩容那次复制是 O(n)，但**平摊**到每次操作仍是 O(1)。

### 六、可哈希协议：`__hash__` 与 `__eq__`

哈希表机制弄清楚了，现在看它施加的约束。

**为什么要求「哈希值不变」？**

```
1. 插入 "key" → hash=123 → 存入槽位 3
2. "key" 内容变化 → hash 变成 456
3. 查找 "key" → hash=456 → 去槽位 6 → 空的 ✗
4. 数据永久丢失（在槽位 3 但永远找不到）
```

这不是实现限制，而是数学上的不可能——插入和查找用了不同的索引。

Python 哈希协议 = 两个特殊方法：

```
1. __hash__() → 返回整数，用于定位槽位
2. __eq__()   → 比较相等，冲突时做最终判定
3. 核心契约：a == b → hash(a) == hash(b)
4. 哈希值在对象生命周期内不变
```

### 七、可变 vs 不可变：可哈希性判断

| 类型 | 可变？ | 可哈希？ | dict 键/set 元素？ |
|------|--------|----------|-------------------|
| `int`, `float`, `str`, `bytes` | ❌ | ✅ | ✅ |
| `tuple`（元素全可哈希） | ❌ | ✅ | ✅ |
| `frozenset` | ❌ | ✅ | ✅ |
| `list` | ✅ | ❌ | ❌ |
| `dict` | ✅ | ❌ | ❌ |
| `set` | ✅ | ❌ | ❌ |
| `tuple`（包含 list） | ❌ | ❌ | ❌ |
| 自定义类（默认） | ✅ | ✅（基于 id） | ✅ |

> ⚠️ tuple 本身不可变，但包含可变子元素时整体不可哈希。可哈希性要求**递归的不可变性**——整个对象图都必须不可变。

### 八、自定义类的可哈希性

**默认**：基于 `id`，永远可哈希（不定义 `__eq__` 时）。

```python
class Person:
    def __init__(self, name):
        self.name = name

p1 = Person("Alice")
p2 = Person("Alice")
print(hash(p1) == hash(p2))  # False（不同对象）
```

**自定义值语义**：定义 `__eq__` + `__hash__`，两者必须配套。

```python
class Person:
    def __init__(self, name, age):
        self.name = name
        self.age = age

    def __eq__(self, other):
        if not isinstance(other, Person):
            return NotImplemented
        return self.name == other.name and self.age == other.age

    def __hash__(self):
        return hash((self.name, self.age))  # 基于不可变属性

people = {Person("Alice", 30), Person("Alice", 30)}
print(len(people))  # 1（自动去重）
```

> ⚠️ 定义 `__eq__` 而不定义 `__hash__` → Python 自动将 `__hash__` 设为 `None` → 对象不可哈希。`__eq__` 基于可变属性 → 不应定义 `__hash__`。

## 📎 例子

### 哈希表操作

```python
# 关键操作：index = hash(key) % table_size
index = hash("username") % 16  # 确定键存在哪个槽位
```

### 各类型的可哈希性

```python
# ✅ 可哈希
hash("hello")               # 字符串
hash((1, 2, 3))             # 全不可变元素的元组
hash(frozenset([1,2,3]))    # 冻结集合

# ❌ 不可哈希
hash([1, 2, 3])             # TypeError: unhashable type: 'list'
hash({"key": "value"})      # TypeError: unhashable type: 'dict'
hash((1, [2, 3]))           # TypeError: 包含可变元素的元组
```

### 表满 vs 有空槽

```python
# 表满：失败查找退化为 O(n)
hash("xyz") → 槽位 3，被占 → 槽位 4，被占 → ... → 走完全部 → 确认不存在（O(n)）

# 有空槽：失败查找随时刹车
hash("xyz") → 槽位 3，被占 → 槽位 4，空 ✓ → 确认不存在（O(1)）
```

**空槽 = 探测的终止符**。Python 在 α≈2/3 时扩容，表里永远保留 ≥1/3 的空槽。

### 扩容的平摊分析

负载因子达到 2/3 → 表翻倍 → 所有元素重新哈希到新表（O(n) 单次）→ 但平摊到每次插入仍是 O(1)，和 `list.append` 同理。

## ⚡ 反例 / 边界

### 1. 表满导致 O(n) 退化

表满时每次失败查找要探测所有槽位才能确认 key 不存在，O(1) 退化为 O(n)。Python 通过 2/3 负载因子阈值杜绝此情况。

### 2. 可变对象作为字典键

```python
d = {}
d[[1, 2]] = "value"         # TypeError: list 不可哈希

t = (1, [2, 3])
d[t] = "value"              # TypeError: tuple 包含可变 list
```

### 3. 可变属性用于 `__hash__` 的陷阱

```python
class Broken:
    def __init__(self, items):
        self.items = items
    def __eq__(self, other):
        return self.items == other.items
    def __hash__(self):
        return hash(tuple(self.items))  # 基于可变属性！

obj = Broken([1, 2, 3])
d = {obj: "value"}
obj.items.append(4)          # 哈希值改变！
# d[obj] → KeyError          数据还在但找不到了
```

### 4. 定义 `__eq__` 没定义 `__hash__`

```python
class Bad:
    def __eq__(self, other): return True
    # Python 自动设 __hash__ = None

d = {Bad(): 1}  # TypeError: unhashable type: 'Bad'
```

### 5. 哈希洪水攻击（HashDoS）

攻击者构造大量哈希冲突的 key 使所有元素堆积到同一槽位，O(1)→O(n)。Python 3.4+ 使用 SipHash + 随机种子防御。

### 6. 好的哈希函数 ≠ 好的哈希表索引

Java HashMap 额外使用 `h ^ (h >>> 16)` 扰动——即使哈希函数均匀，也要对哈希值做二次扰动增强低位随机性（取模看低位）。
