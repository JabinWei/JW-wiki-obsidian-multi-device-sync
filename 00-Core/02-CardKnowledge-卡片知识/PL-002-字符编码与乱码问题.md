# 字符编码与乱码问题

## 💡 核心概念

计算机只能存储二进制（0 和 1），**字符编码（Character Encoding）** 就是「字符」与「二进制数字」之间的映射规则。理解编码是解决一切乱码问题的钥匙。

- **编码（Encode）**：字符 → 二进制字节，用于存储和传输。
- **解码（Decode）**：二进制字节 → 字符，用于人类阅读。
- **乱码（Mojibake）**：用了**错误的编码方式去解码**，就像用中文词典去查英文单词。

> 🔑 核心原则：**用什么编码写的，就必须用什么编码去读。**

---

## 🔍 详细说明

### 一、为什么需要编码？—— 从一根电话线说起

早期的计算机只处理英文，**ASCII**（American Standard Code for Information Interchange）用一个字节（8 位）中的低 7 位来表示 128 个字符：英文字母、数字、标点符号和控制字符。

| 编码 | 字节数 | 字符数 | 适用场景 |
|------|--------|--------|----------|
| **ASCII** | 1 字节 | 128 | 纯英文 |
| **Latin-1 (ISO-8859-1)** | 1 字节 | 256 | 西欧语言（在 ASCII 基础上扩展了 128 个字符） |
| **GB2312 / GBK** | 1-2 字节 | 数千/数万 | 简体中文 |
| **Shift-JIS** | 1-2 字节 | 数千 | 日文 |
| **UTF-8** | 1-4 字节 | 全人类所有字符 | **互联网标准，推荐所有场景** |
| **UTF-16** | 2 或 4 字节 | 全人类所有字符 | Windows 内部、Java 内部 |
| **UTF-32** | 4 字节 | 全人类所有字符 | 极少使用（浪费空间） |

问题来了：每种语言各搞一套编码，互不兼容。一封中文邮件在日文系统上打开 → 乱码。这时候需要一个「大一统」方案。

### 二、Unicode：所有字符的「身份证号」

**Unicode** 不是编码，而是一个**字符集**——它为世界上几乎每一个字符分配了一个唯一的数字编号，称为**码点（Code Point）**，记作 `U+XXXX`。

```
A  → U+0041
中 → U+4E2D
😀 → U+1F600
```

但 Unicode 只定义了「哪个数字代表哪个字符」，**不定义怎么存成字节**。把这个数字变成字节的任务，就交给了具体的编码方案（UTF-8、UTF-16 等）。

> 💡 **类比**：Unicode = 全国身份证号（每个人有唯一编号），UTF-8 = 一种具体的存储和传输此编号的格式。

### 三、UTF-8：互联网的事实标准

**UTF-8**（Unicode Transformation Format - 8 bit）是目前使用最广泛的编码。核心设计：

- **变长编码**：每个字符占用 1~4 个字节。
  - ASCII 字符（英文、数字、标点）→ **1 字节**，与 ASCII 完全兼容。
  - 拉丁字母、希腊字母等 → **2 字节**。
  - 中文、日文、韩文 → **3 字节**（大部分常用汉字）。
  - 表情符号、罕见汉字 → **4 字节**。

- **向后兼容 ASCII**：一个纯 ASCII 文本同时也是一个合法的 UTF-8 文本。这是它取代其他编码的最重要原因。

- **无字节序问题**：UTF-8 的设计天然避免了「大端/小端」的争议。

**为什么推荐 UTF-8？**

| 优势 | 说明 |
|------|------|
| **通用** | 能表示所有 Unicode 字符，一份文件走天下 |
| **兼容** | 完全兼容 ASCII，不破坏任何已有的英文系统 |
| **节省空间** | 英文只占 1 字节，不像 UTF-16 每个字符至少 2 字节 |
| **无 BOM 争议** | 不像 UTF-16 需要 BOM 来标识字节序 |

### 四、UTF-16 和 BOM

**UTF-16** 用 2 或 4 个字节表示一个字符。它是 Java 和 Windows 内部使用的编码。

**BOM（Byte Order Mark，字节序标记）** 是 UTF-16 的必需品：文件开头的一个特殊标记（`U+FEFF`），告诉程序「我是大端还是小端」。

UTF-8 也可以加 BOM（`EF BB BF`），但**不推荐**——Linux/Mac 上的很多工具会把它当成普通字符，导致文件开头出现一个不可见的奇怪符号。

### 五、各语言中如何处理编码

**Python：**

```python
# 编码：str → bytes
text = "你好"
b = text.encode("utf-8")   # b'\xe4\xbd\xa0\xe5\xa5\xbd'

# 解码：bytes → str
b = b'\xe4\xbd\xa0\xe5\xa5\xbd'
text = b.decode("utf-8")   # '你好'

# 文件读写：务必显式指定 encoding
with open("file.txt", "r", encoding="utf-8") as f:
    content = f.read()
```

**Java：**

```java
// String 内部使用 UTF-16
String s = "你好";
byte[] utf8Bytes = s.getBytes(StandardCharsets.UTF_8);
String restored = new String(utf8Bytes, StandardCharsets.UTF_8);

// 文件读写
BufferedReader reader = new BufferedReader(
    new InputStreamReader(new FileInputStream("file.txt"), StandardCharsets.UTF_8)
);
```

**JavaScript/Node.js：**

```javascript
// 编码
const encoder = new TextEncoder();  // 默认 UTF-8
const bytes = encoder.encode("你好");

// 解码
const decoder = new TextDecoder("utf-8");
const text = decoder.decode(bytes);

// Node.js Buffer
Buffer.from("你好", "utf-8");
buf.toString("utf-8");
```

**Go：**

```go
// Go 源码就是 UTF-8，string 底层是 UTF-8 字节序列
s := "你好"
// 遍历 rune（Unicode 码点）而非字节
for _, r := range s {
    fmt.Printf("%c -> U+%X\n", r, r)
}
```

## 📎 例子

### 乱码的产生与排查

**场景：Windows 上用记事本写了一个文件，在 Mac 上打开乱码了。**

真相：Windows 记事本默认用 **GBK** 保存中文，Mac 的文本编辑器默认用 **UTF-8** 打开。编码不匹配 → 乱码。

**排查思路（通用）：**

1. **确定「写入时的编码」**：文件是在什么系统、用什么工具创建的？
2. **确定「读取时的编码」**：你现在用的工具默认用什么编码打开？
3. **尝试常见编码解码**：

```python
# 如果你的文件疑似 GBK 编码，尝试：
with open("file.txt", "r", encoding="gbk") as f:
    print(f.read())
```

4. **如果文件来源不明，用 `chardet` 自动检测**：

```python
import chardet
with open("file.txt", "rb") as f:
    result = chardet.detect(f.read())
    print(result)  # {'encoding': 'gb2312', 'confidence': 0.99}
```

### 编码转换实战

```python
# 场景：读取一个 GBK 编码的文件，保存为 UTF-8
with open("gbk_file.txt", "r", encoding="gbk") as fin:
    content = fin.read()

with open("utf8_file.txt", "w", encoding="utf-8") as fout:
    fout.write(content)
```

## ⚡ 反例 / 边界

### Python 中常见的「编码坑」

**坑 1：Windows 上不指定 encoding**

```python
# ❌ 危险：Windows 默认用 GBK，UTF-8 文件会乱码
with open("data.txt", "r") as f:
    content = f.read()

# ✅ 正确：始终显式指定
with open("data.txt", "r", encoding="utf-8") as f:
    content = f.read()
```

**坑 2：`bytes.decode()` 和 `str.encode()` 编解码不匹配**

```python
b = "中文".encode("utf-8")      # 用 UTF-8 编码
text = b.decode("gbk")           # ❌ 用 GBK 解码 → 乱码
```

**坑 3：中文字符的字节长度错觉**

```python
s = "Hello中文"
print(len(s))              # 7（7 个字符）
print(len(s.encode("utf-8")))  # 12（5 + 7 = 12 字节，中文各 3 字节 + 英文 5 字节 + "中文" 6 字节）

# 注意：len("中文") = 2 字符，但 UTF-8 编码后是 6 字节！
```
